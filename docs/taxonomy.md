# v1 诊断 Taxonomy

> 系统诊断引擎使用的 **25 个症状标签**完整规格。
>
> 锁定日：2026-07-06 · v1 · 待端到端验证后复审

## 1. 设计原则

1. **症状驱动**（不是概念）：标签是"用户描述自己时的语言"。例 `拖延`（不是 `任务启动障碍`），`信念冲突`（不是 `限制性信念`）。
2. **口语化**：用户用自然语言说出来时，应该用得上。
3. **互斥不强**：两个标签可同时点亮（如 `信念冲突` + `易怒`），由 LLM 自由选 1-3 个。
4. **KB 服务导向**：每个标签至少要有 1 个 K-id 覆盖（v1 现状：25/25 全覆盖，其中 7 个标签覆盖率 ≤ 2 条，需日后扩 KB）。

## 4. LLM 标签匹配规则

诊断 prompt (`docs/prompt.md`) 中规定的 4 条匹配规则：

1. **用户语言 → 标签**，不是「反向匹配 KB 概念」。例：用户说「我做不下去怕不够好」→ 匹配 `完美主义`。
2. **宁少勿真**：不确定的少选 / 不选。`diagnoses: []` 是合法输出。
3. 每诊断必有 `evidence`，引用用户原话的具体短语。
4. 最多选 3 个标签。

## 5. 扩展

### 加新标签

何时需要：
- 新的诊断场景出现，25 个 v1 全不命中
- 用户反馈"某个状态没被识别"

不要：
- 加学术标签（如 `前额叶失活`）
- 加太接近的标签（如同时有 `焦虑` 和 `担忧`，拆出 `担忧`）
- 同时动太多（每次 +1，复盘 1 个月再 +1）

### 步骤

1. 在 [taxonomy.md](taxonomy.md) 表里加行（写在原表格下方或加新分类）
2. 在所有 KB JSON 里给该标签至少 3-5 条 K-id 覆盖
3. 跑 `python3 scripts/relabel_kb.py` 不会自动添加新标签映射，需手改 [scripts/relabel_kb.py](../scripts/relabel_kb.py) 里的 MAPPING
4. 跑 `python3 scripts/build_kb_index.py` 和 `python3 scripts/build_prompt.py` 让索引和 prompt 更新
5. 跑 [scripts/test_e2e.py](../scripts/test_e2e.py) 至少 2 个验证场景
6. 写 changelog（`docs/changelog.md`）

### 减标签

更慎重。一旦减，要更新：
- taxonomy 文档
- 所有 KB JSON 里 K-id 的 diagnosis_tags
- prompt-v1.md（`build_prompt.py` 的 CATEGORY_ORDER）
- relabel_kb.py 的 V1_TAGS

不建议在没有重大失败证据前减标签。
